# Idea Gauntlet Skill

Run any idea, feature, strategy, or decision through five pressure-testing techniques. Each technique attacks the idea from a different angle. Ideas that survive the full gauntlet are ready to execute with confidence.

## When to Use

- Before building a new feature in the WC App
- Before expanding into a new market
- Before committing budget to a campaign or initiative
- Before changing a process or workflow
- Before making a strategic hire or partnership
- Anytime someone says "I have an idea" and you want to stress-test it

## Technique 1 — Destroy & Rebuild

**Purpose:** Find the fatal flaw before it finds you.

### Process

1. **State the idea clearly** in one sentence
2. **Destroy it** — list every reason it could fail, every objection, every risk. Be ruthless. Aim for 7-10 destruction points:
   - Why would customers NOT want this?
   - What could go wrong operationally?
   - What does the competition do better?
   - What assumptions are we making that might be wrong?
   - What is the worst-case scenario?
   - How could this waste money or time?
   - Who would actively resist this?
3. **Rank destructions** by severity (1 = minor annoyance, 5 = kills the idea)
4. **Rebuild** — for every severity 4-5 destruction, propose a specific mitigation or pivot
5. **Verdict:** Does the rebuilt version survive? Is it stronger?

### Output Format

```
DESTROY & REBUILD: [Idea Name]

IDEA: [One sentence]

DESTRUCTIONS:
1. [Destruction] — Severity: X/5
2. [Destruction] — Severity: X/5
...

REBUILDS (for severity 4-5 only):
- [Destruction] → [Mitigation/Pivot]
- [Destruction] → [Mitigation/Pivot]

VERDICT: [SURVIVES / NEEDS PIVOT / KILL]
```

## Technique 2 — Three Eyes

**Purpose:** See the idea from every stakeholder perspective.

### Process

Evaluate the idea through three lenses:

**Eye 1 — The User/Customer**
- What problem does this solve for them?
- Would they pay for this (with money, time, or attention)?
- What is their current alternative?
- How would they describe this to a friend?

**Eye 2 — The Builder/Operator**
- How hard is this to build/implement?
- What is the maintenance burden?
- Does this create technical debt?
- Do we have the skills and resources?

**Eye 3 — The Business/Strategy**
- Does this move a key metric?
- What is the ROI timeline?
- Does this create a competitive moat?
- Does this align with the company's 12-month priorities?

### Output Format

```
THREE EYES: [Idea Name]

USER EYE:
- Solves: [problem]
- Willingness to adopt: [High/Medium/Low]
- Current alternative: [what they do today]
- Would they tell a friend: [Yes/No — why]

BUILDER EYE:
- Build complexity: [1-5]
- Maintenance burden: [1-5]
- Tech debt risk: [1-5]
- Team capability: [Ready/Need Hire/Need Training]

BUSINESS EYE:
- Key metric impact: [which metric, estimated lift]
- ROI timeline: [weeks/months/quarters]
- Moat creation: [Yes/No — why]
- Strategic alignment: [High/Medium/Low]

CONSENSUS: [All three eyes agree / Eyes conflict on X]
```

## Technique 3 — Compression

**Purpose:** Strip the idea to its absolute minimum viable version.

### Process

1. **List every feature/component** of the idea as proposed
2. **For each, ask:** "Would the idea still work without this?" — mark as CORE or NICE-TO-HAVE
3. **Compress to the minimum:** What is the smallest possible version that delivers the core value?
4. **Time-box it:** Can the compressed version be built/shipped in one week?
5. **Define the test:** What metric or observation would prove this compressed version works?

### Output Format

```
COMPRESSION: [Idea Name]

FULL VERSION:
- [Feature 1] — CORE / NICE-TO-HAVE
- [Feature 2] — CORE / NICE-TO-HAVE
- [Feature 3] — CORE / NICE-TO-HAVE
...

COMPRESSED VERSION:
[Describe the minimum viable version in 2-3 sentences]

ONE-WEEK BUILD: [Yes/No — if No, compress further]

SUCCESS TEST:
- Metric: [what to measure]
- Target: [what number = success]
- Timeline: [how long to run the test]
```

## Technique 4 — Assumption Hunt

**Purpose:** Surface and validate the hidden assumptions that make or break the idea.

### Process

1. **List every assumption** the idea relies on to succeed (aim for 5-10)
   - "We assume customers want X"
   - "We assume we can build this in Y time"
   - "We assume the market size is Z"
   - "We assume competitors won't react"
   - "We assume the unit economics work at scale"
2. **Classify each** as:
   - VALIDATED — we have data or evidence
   - TESTABLE — we can validate before committing
   - LEAP OF FAITH — we are guessing and cannot easily test
3. **For each TESTABLE assumption**, define a cheap/fast validation method
4. **For each LEAP OF FAITH**, decide if the risk is acceptable

### Output Format

```
ASSUMPTION HUNT: [Idea Name]

ASSUMPTIONS:
1. [Assumption] — VALIDATED (evidence: [what])
2. [Assumption] — TESTABLE (test: [how to validate cheaply])
3. [Assumption] — LEAP OF FAITH (risk level: [1-5])
...

CRITICAL LEAPS OF FAITH:
- [Assumption] — Acceptable risk? [Yes/No — why]

VALIDATION PLAN:
Before committing, test these first:
1. [Assumption] — Test: [method] — Timeline: [X days]
2. [Assumption] — Test: [method] — Timeline: [X days]
```

## Technique 5 — Time Traveler

**Purpose:** Project the idea forward in time to see its long-term consequences.

### Process

Imagine the idea has been implemented. Evaluate it at three time horizons:

**+1 Month**
- What has changed?
- What early signals (positive or negative) are visible?
- What unexpected problems have appeared?

**+6 Months**
- Is this still being used/maintained?
- Has it created any second-order effects?
- Would we build it again knowing what we know?

**+2 Years**
- Is this a core part of the business or a relic?
- Did it compound (grow in value) or decay?
- What did we miss that we wish we had thought about?

### Output Format

```
TIME TRAVELER: [Idea Name]

+1 MONTH:
- Status: [Thriving/Struggling/Dead]
- Signals: [what we would observe]
- Surprises: [what we didn't expect]

+6 MONTHS:
- Status: [Core/Maintained/Abandoned]
- Second-order effects: [ripple effects]
- Would rebuild: [Yes/No — why]

+2 YEARS:
- Status: [Compounding/Stable/Decaying/Gone]
- Missed considerations: [what we wish we had planned for]
- Legacy impact: [positive or negative]

TIME VERDICT: [Idea ages well / Idea has a shelf life / Idea is disposable]
```

## Full Gauntlet Order

When running the full gauntlet, execute in this order:

1. **Destroy & Rebuild** — Find the flaws
2. **Three Eyes** — Check all perspectives
3. **Compression** — Find the minimum viable version
4. **Assumption Hunt** — Surface hidden risks
5. **Time Traveler** — Project forward

After all five, deliver a **Final Verdict**:

```
GAUNTLET FINAL VERDICT: [Idea Name]
====================================

Survived: [X/5 techniques]

GO / NO-GO / PIVOT

[2-3 sentence summary of why]

IF GO:
- Start with: [compressed version from Technique 3]
- Validate first: [top assumptions from Technique 4]
- Watch for: [risks from Technique 1]

IF PIVOT:
- Pivot to: [rebuilt version from Technique 1]
- Because: [reason]

IF NO-GO:
- Kill because: [fatal flaw]
- Alternative: [better approach to the same underlying need]
```

## Quality Standards

- Every technique must be completed with specific, detailed analysis — not vague hand-waving
- Use real numbers, real names, real scenarios wherever possible
- Do not soft-pedal. The point is to pressure-test, not validate
- If an idea is bad, say so clearly and offer an alternative
- If an idea is great, say so and highlight what makes it strong

## The Meta-Rule

The gauntlet exists to make ideas better, not to kill them. The goal is to find the version of the idea that survives every attack. Most ideas have a strong core hidden under weak assumptions and missing details. The gauntlet strips away the weak parts and reveals the strong core.

If an idea survives all five techniques, build it with full confidence.
If an idea fails, it saved you from wasting weeks or months on something that would not have worked.
Either outcome is a win.
